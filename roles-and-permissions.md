# Roles & Permissions — LMS Платформа обучения диспетчеров

---

## 1. Цели по ролям

### Студент
Пройти курс обучения от первой главы до последней, сдать тест и устный экзамен по каждой главе, получить подтверждение о завершении обучения. Студент движется по строго последовательному пути: урок → тест → экзамен → следующая глава. Он не управляет контентом и не видит чужие данные.

### Менеджер
Контролировать прогресс своих студентов, принимать устные экзамены и выносить решения (PASS / RETRY / DISBAND). Менеджер отвечает за качество подготовки своей группы, но не управляет контентом курса и не видит студентов других менеджеров.

### Администратор
Управлять платформой целиком: контент, пользователи, аналитика. Единственная роль, которая может создавать аккаунты, редактировать курс, назначать менеджеров студентам и просматривать агрегированную статистику по всей платформе.

## 3. Действия в системе по ролям

### Студент — полный список действий

**Аутентификация:**
- Войти в систему (email + пароль)
- Обновить собственный профиль (имя, пароль)
- Подписаться на push-уведомления

**Обучение:**
- Просмотреть доступные главы курса
- Открыть урок (только если разблокирован)
- Пройти урок и нажать «Отметить как пройденный»
- Перейти к следующему уроку (открывается автоматически после завершения предыдущего)

**Тест:**
- Пройти тест по завершённой главе (порог 80%)
- Просмотреть результат своей попытки
- Перепройти тест при неудаче (количество попыток не ограничено)

**Экзамен:**
- Запросить устный экзамен (только после сдачи теста)
- Просмотреть статус запроса и решение менеджера
- Прочитать комментарий менеджера к решению
- Получить push-уведомление о результате экзамена

**Фидбек:**
- Оставить текстовый отзыв на урок (один раз на урок)

**Прогресс:**
- Просмотреть общий прогресс по курсу (% завершения)
- Просмотреть статус каждой главы (заблокирована / в процессе / завершена)

---

### Менеджер — полный список действий

**Аутентификация:**
- Войти в систему
- Обновить собственный профиль

**Студенты:**
- Просмотреть список назначенных студентов
- Просмотреть детальный прогресс каждого студента
- Просмотреть результаты тестов студента (все попытки, все главы)
- Просмотреть историю экзаменов студента

**Экзамены:**
- Получить push-уведомление о новом запросе на экзамен
- Просмотреть очередь ожидающих экзаменов (только свои студенты)
- Назначить время экзамена
- Вынести решение: PASS / RETRY / DISBAND
- Оставить обязательный комментарий к решению (минимум 10 символов)
- Просмотреть историю всех своих решений

**Аналитика группы:**
- Просмотреть сводку по группе: средний балл, % завершения, количество DISBAND

---

### Администратор — полный список действий

**Пользователи:**
- Создать аккаунт (студент, менеджер, другой администратор)
- Назначить менеджера студенту
- Активировать / деактивировать аккаунт
- Сбросить пароль
- Просмотреть всех пользователей с фильтрацией по роли

**Контент:**
- Создать / редактировать / удалить курс
- Создать / редактировать / удалить главу
- Создать / редактировать / удалить урок (тип, контент, порядок)
- Создать / редактировать / удалить вопросы теста и варианты ответов
- Опубликовать / снять с публикации курс или главу
- Изменить порядок глав и уроков
- Изменить порог прохождения теста (по умолчанию 80%)

**Экзамены:**
- Просмотреть все экзамены на платформе
- Отменить / изменить решение менеджера (с обязательным комментарием причины)
- Повторно активировать студента после DISBAND

**Аналитика:**
- Просмотреть дашборд: всего студентов, активных, средний балл, количество ожидающих экзаменов
- Детальная аналитика по любому студенту
- Аналитика по менеджерам (количество PASS/RETRY/DISBAND)
- Экспорт данных в CSV

**Система:**
- Просмотреть audit log
- Управлять конфигурацией платформы

## 4. RBAC-модель

### Принципы

Система использует плоскую RBAC (Role-Based Access Control) с тремя ролями и ownership-проверками на уровне сервиса. Роли не наследуются (менеджер не включает права студента, и так далее). Каждый endpoint проверяет: 1) аутентификацию, 2) роль, 3) ownership (принадлежит ли ресурс текущему пользователю).

### Структура

```
Permission = Action × Resource × Scope
```

Где **Scope** бывает трёх видов:
- `OWN` — только свои данные
- `ASSIGNED` — данные назначенных студентов (для менеджера)
- `ALL` — все данные на платформе (только для admin)

### Именование разрешений

```
<resource>:<action>:<scope>

Примеры:
  lesson:read:own
  lesson:complete:own
  exam:review:assigned
  exam:disband:assigned
  analytics:read:all
  user:create:all
```

### Реализация в NestJS

Текущая реализация использует глобальный `JwtAuthGuard` + глобальный `RolesGuard` с декоратором `@Roles()`. Ownership-проверки реализуются внутри каждого сервиса.

```typescript
// Пример: только менеджер видит экзамены, и только своих студентов
@Get('pending')
@Roles(UserRole.MANAGER, UserRole.ADMIN)
async getPendingExams(@CurrentUser() user: User) {
  return this.examsService.getPendingForManager(user.id, user.role);
}

// В сервисе — ownership scope
async getPendingForManager(managerId: string, role: UserRole) {
  const where = role === UserRole.ADMIN
    ? { status: ExamStatus.REQUESTED }
    : { status: ExamStatus.REQUESTED, managerId }; // ASSIGNED scope
  return this.prisma.examRequest.findMany({ where });
}
```

## 5. Действия, подлежащие логированию (Audit Log)

Все события пишутся в отдельную таблицу `AuditLog` со структурой:

```
id, actorId (кто), actorRole, action, resourceType, resourceId, payload (JSON), ip, userAgent, createdAt
```

### Критические события (логировать всегда)

| Событие | Кто инициирует | Зачем |
|---|---|---|
| Вход в систему (успешный) | Любой | Безопасность |
| Вход в систему (неуспешный) | Любой | Выявление брутфорса |
| Смена пароля | Любой | Безопасность |
| Создание аккаунта | Админ | Соответствие |
| Деактивация аккаунта | Админ | Аудит изменений |
| Назначение менеджера студенту | Админ | Прослеживаемость |
| Решение PASS | Менеджер / Админ | Ключевое бизнес-событие |
| Решение RETRY | Менеджер / Админ | Ключевое бизнес-событие |
| Решение DISBAND | Менеджер / Админ | Высокий риск: деактивация студента |
| Отмена / изменение решения | Админ | Должна быть причина |
| Повторная активация студента после DISBAND | Админ | Высокий риск |

### Важные бизнес-события (логировать)

| Событие | Кто инициирует | Зачем |
|---|---|---|
| Завершение урока | Студент | Прогресс |
| Прохождение теста (PASS / FAIL + балл) | Студент | Результат |
| Запрос экзамена | Студент | Триггер для менеджера |
| Назначение времени экзамена | Менеджер | Договорённость |
| Публикация / снятие с публикации курса | Админ | Изменение контента |
| Создание / удаление вопроса | Админ | Изменение теста |
| Изменение порога прохождения теста | Админ | Влияет на всех студентов |

### Не логировать

- Просмотр урока (только чтение, нет изменений)
- Обновление собственного профиля (имя)
- Просмотр прогресса
- Push-уведомления

### Пример записи в audit log

## 6. Пограничные случаи и правила

**Менеджер не может видеть чужих студентов.** Если студент не назначен конкретному менеджеру (`managerId IS NULL`), он невидим для менеджеров. Только админ видит таких студентов.

**Решение DISBAND необратимо без участия администратора.** После DISBAND студент деактивируется (`isActive = false`). Только администратор может вернуть студента, повторно активировав аккаунт. Менеджер не может отменить своё решение.

**Комментарий к решению — обязателен.** Любое решение (PASS / RETRY / DISBAND) без комментария отклоняется на уровне DTO-валидации (`@MinLength(10)`). Это защищает студента от немотивированных решений.

**Студент не может запросить два экзамена по одной главе одновременно.** Если уже есть запрос со статусом `REQUESTED` или `SCHEDULED` по данной главе, новый запрос возвращает `409 Conflict`.

**Тест можно перепроходить неограниченно.** Засчитывается последняя попытка, а не лучшая — это мотивирует студента к вдумчивой подготовке, а не к угадыванию.

**Администратор не должен проводить экзамены в обход менеджера в штатном режиме.** Технически право есть (`exam:review:all`), но в audit log будет явно видно, что решение принял admin, а не назначенный менеджер.
